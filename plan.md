# BIR Function Signature Byval Metadata Text Retirement Runbook

Status: Active
Source Idea: ideas/open/191_bir_function_signature_byval_metadata_text_retirement.md
Activated After: ideas/closed/190_lir_call_argument_structured_payload_boundary.md

## Purpose

Retire the remaining metadata-rich BIR function-signature path that parses
rendered `signature_text` to decide explicit byval aggregate parameter
handling.

## Goal

Make structured function signature metadata the authority for generated byval
aggregate parameter layout and ABI facts, while keeping `signature_text` as
final output spelling and legacy no-metadata compatibility input.

## Core Rule

When structured signature metadata is available, BIR lowering must not parse
`signature_text` to recover semantic byval facts. Text parsing may remain only
behind an explicit raw/no-metadata compatibility boundary.

## Current Targets

- `BirFunctionLowerer::collect_aggregate_params`.
- Function signature metadata producers for `LirFunction::signature_params` and
  `signature_param_type_refs`.
- Any helper that turns structured function signature facts into
  `ParsedFunctionSignatureParam` values.
- Tests proving stale rendered signature text cannot drive generated byval
  aggregate handling when structured metadata exists.

## Non-Goals

- Do not redesign the full call ABI.
- Do not remove `signature_text` as final printed output.
- Do not change raw hand-built/no-metadata LIR fixture behavior except to make
  its fallback boundary explicit.
- Do not rework unrelated aggregate local-slot lowering.
- Do not claim progress through printer-only assertions, expectation
  downgrades, or helper renames that leave the same text authority in place.

## Working Model

- Idea 190 closed the structured call-argument payload boundary, so idea 191 can
  focus on function signature byval metadata rather than call-site argument
  text.
- Generated LIR already exposes structured signature params and type refs, but
  `collect_aggregate_params` still has a path that parses `signature_text` to
  detect explicit byval spelling.
- The desired route is to add or reuse structured byval metadata at the
  function signature boundary, then make aggregate parameter collection fail
  closed for metadata-rich functions when that metadata is stale or missing.
- Legacy no-metadata LIR fixtures may still parse rendered signature text, but
  that compatibility path must be visibly separate from generated metadata-rich
  lowering.

## Step 1: Inventory Function Signature Byval Authority

Goal: identify where generated function-signature byval handling still depends
on rendered `signature_text`.

Concrete actions:
- Inspect `BirFunctionLowerer::collect_aggregate_params`.
- Trace `has_structured_signature_params`, `structured_signature_params`, and
  `parse_function_signature_params`.
- Trace producers of `LirFunction::signature_params` and
  `signature_param_type_refs` for aggregate/byval declarations and definitions.
- Separate generated metadata-rich paths from legacy hand-built/no-metadata LIR
  fixtures.
- Record the exact first implementation target and the compatibility fallback
  boundary in `todo.md`.

Completion check:
- `todo.md` names the text-authority site, the structured metadata producer or
  gap, and the first code slice for retiring metadata-rich `signature_text`
  byval parsing.

## Step 2: Add Or Select Structured Byval Signature Metadata

Goal: represent explicit byval state directly in structured function signature
metadata.

Concrete actions:
- Add or reuse a field at the existing LIR function signature boundary for
  explicit byval aggregate parameters.
- Keep the metadata aligned with `signature_params` and
  `signature_param_type_refs`.
- Preserve `signature_text` as output spelling and no-metadata compatibility
  input.
- Update verifier checks so stale or inconsistent metadata-rich byval facts are
  rejected instead of recovered from text.

Completion check:
- Generated byval function signatures carry structured explicit-byval facts
  without requiring `signature_text` parsing to discover them.

## Step 3: Route Aggregate Param Collection Through Structured Facts

Goal: make structured signature metadata the semantic authority for aggregate
parameter collection when available.

Concrete actions:
- Update `collect_aggregate_params` and related helpers to consume structured
  byval metadata for metadata-rich functions.
- Keep `parse_function_signature_params(function_.signature_text)` reachable
  only for explicit no-metadata compatibility cases.
- Make missing or stale structured byval metadata fail closed for generated
  metadata-rich functions.
- Preserve existing aggregate layout behavior for valid structured metadata.

Completion check:
- Metadata-rich aggregate parameter collection no longer parses
  `signature_text` to decide explicit byval handling.

## Step 4: Prove Stale Signature Text Cannot Override Metadata

Goal: add focused tests that fail if rendered `signature_text` still controls
generated byval aggregate semantics.

Concrete actions:
- Add or update BIR aggregate/function-signature tests with stale byval text.
- Include a positive structured byval case and a metadata-rich stale/missing
  metadata rejection case.
- Keep a legacy no-metadata fallback test if text parsing remains supported
  there.
- Avoid tests that only inspect final printed headers.

Completion check:
- Focused tests prove structured byval metadata controls generated aggregate
  parameter handling and raw/no-metadata fallback is explicit.
