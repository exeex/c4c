# Route 3 Memory/Source Oracle Or Printer Row Runbook

Status: Active
Source Idea: ideas/open/208_route3_memory_source_oracle_printer_row.md

## Purpose

Transcribe the Route 3 memory/source diagnostic follow-up into an implementation runbook for exactly one helper-oracle status row or one prepared addressing printer row.

## Goal

Replace one named Route 3 memory/source helper-oracle status row or one prepared addressing printer row with route-native evidence while preserving prepared diagnostic authority, target-addressing fallback, and byte-stable output.

## Core Rule

Route 3 may provide evidence for the selected diagnostic row only when it reproduces the current prepared positive, negative, ambiguity, mismatch, fallback, and target-policy-sensitive behavior. Prepared diagnostics remain authoritative for every non-agreement or unsupported path.

## Current Targets

- One named Route 3 memory/source helper-oracle status row, or one prepared addressing printer row.
- Existing Route 3 memory/source identity or evidence surfaces.
- The prepared helper-oracle or prepared addressing printer behavior currently producing the selected row.
- Tests or expected-output checks covering positive, absent, invalid, duplicate/conflict, mismatch, fallback, target-addressing-sensitive output, and byte-stable printer/debug text.

## Non-Goals

- Do not migrate whole memory oracle families, printer sections, target wrappers, public fallback, aggregate lookup behavior, or `PreparedFunctionLookups`.
- Do not move target-addressing policy, address formation, materialization, final operands, or fallback-sensitive output into Route 3.
- Do not delete prepared APIs, retire prepared diagnostics, rewrite expected strings, weaken statuses, refresh baselines, or claim helper renames as progress.
- Do not broaden this plan to semantic-reader migration from prior Route 3 work.

## Working Model

- The selected row must have a prepared answer that remains available for comparison and fallback.
- Route-native evidence can replace the row only when it matches the prepared diagnostic contract for that row.
- Absent Route 3 facts, non-memory facts, invalid references, duplicate or ambiguous evidence, alias/address conflicts, route/prepared mismatches, and target-policy-sensitive cases must preserve prepared behavior.
- Byte-stable output is part of the contract unless a separate semantic proof justifies a string change and the supervisor explicitly accepts it.

## Execution Rules

- Name the exact helper-oracle status row or prepared addressing printer row before implementation.
- Keep code changes local to the selected row and the smallest existing Route 3 evidence adapter needed to prove it.
- Preserve prepared fallback and diagnostic strings before adding route-native acceptance.
- Add or adjust tests only to prove the selected row and fallback matrix; do not refresh expected strings to hide behavior changes.
- For code-changing steps, run a fresh build or compile proof plus the supervisor-delegated narrow Route 3 memory/source proof subset. Escalate broader validation if shared helper-oracle, printer/debug, wrapper, public header, or expected-output surfaces are touched.

## Ordered Steps

### Step 1: Select The Diagnostic Row And Baseline Surface

Goal: Identify the one Route 3 memory/source helper-oracle status row or prepared addressing printer row that can be replaced without expanding diagnostic scope.

Primary targets:
- Current helper-oracle status rows and prepared addressing printer rows for Route 3 memory/source behavior.
- Existing prepared fallback logic for the selected row.
- Existing tests or expected outputs that expose the row and its fallback behavior.

Actions:
- Inspect candidate rows and choose exactly one named row.
- Record whether the selected row is a helper-oracle status row or a prepared addressing printer row.
- Identify the prepared positive status, absent or no-memory status, invalid-reference status, duplicate/conflict behavior, mismatch behavior, and target-addressing fallback surface.
- Identify the narrow proof commands and byte-stability checks needed before implementation.

Completion check:
- `todo.md` names one selected row, its prepared fallback matrix, the Route 3 evidence source to inspect, and the proof subset before implementation begins.

### Step 2: Add Route-Native Evidence Under Prepared Authority

Goal: Reproduce the selected row from Route 3 evidence only when it preserves the prepared diagnostic contract.

Primary targets:
- The selected helper-oracle or prepared addressing printer row from Step 1.
- Existing Route 3 memory/source evidence lookup or agreement helper surfaces.
- The smallest local helper needed to compare Route 3 evidence with prepared diagnostics.

Actions:
- Keep the prepared row-producing path available and authoritative.
- Read Route 3 evidence for the selected row only where the prepared answer can still be used for fallback or comparison.
- Accept route-native replacement only for the same successful diagnostic meaning as prepared.
- Preserve prepared behavior for absent, non-memory, invalid, duplicate/conflict, alias/address conflict, mismatch, fallback, and target-policy-sensitive paths.

Completion check:
- The selected row uses route-native evidence for the positive agreement case, and every non-agreement or unsupported path reports the same prepared status as before.

### Step 3: Prove Fallback Matrix And Byte-Stable Output

Goal: Prove the row replacement is diagnostic evidence work only, not output redesign or target policy migration.

Primary targets:
- Positive row proof for route-native evidence.
- Absent, invalid, duplicate/conflict, mismatch, fallback, and target-policy-sensitive tests or expected-output checks.
- Printer/debug and expected-string surfaces for byte-stable row text.

Actions:
- Add or extend focused tests for the selected row's positive case and fallback matrix.
- Include a target-addressing fallback case if the selected row can be affected by target policy.
- Run the delegated build or compile proof and narrow Route 3 memory/source test subset.
- If output changes unexpectedly, stop and report the attribution instead of refreshing baselines.

Completion check:
- Narrow proof is green, the selected row text is byte-stable, fallback behavior matches prepared diagnostics, and `todo.md` records the exact proof summary.

### Step 4: Finalize Slice State

Goal: Leave lifecycle state ready for supervisor review and commit.

Primary targets:
- `todo.md`
- `test_after.log` when delegated by the supervisor or used as the executor proof artifact.

Actions:
- Update `todo.md` with the selected row, implementation summary, proof commands, and any residual risks.
- Confirm no broad oracle-family, printer-section, wrapper, expected-string, baseline, public fallback, or prepared API migration was made.
- If proof exposes a separate semantic-reader, printer-family, target-wrapper, or policy initiative, record it as follow-up instead of expanding this plan.

Completion check:
- `todo.md` reports the completed packet and proof; the diff remains scoped to one Route 3 memory/source diagnostic row and focused tests.

## Validation

- Minimum acceptance for implementation packets: fresh build or compile proof plus the supervisor-delegated narrow Route 3 memory/source proof subset.
- Required behavioral proof: positive route-native row, absent Route 3 fallback, invalid reference fallback, duplicate/conflict fallback, route/prepared mismatch fallback, and target-addressing fallback when relevant.
- Required non-regression proof: byte-stable helper-oracle or printer/debug row text and no expected-string churn.
- Broader validation is required if a shared Route 3 helper, public prepared header, wrapper path, printer/debug family, target-addressing behavior, or expected-output surface is modified.
