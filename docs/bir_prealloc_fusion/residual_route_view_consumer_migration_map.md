# Residual Route-View Consumer Migration Map

Status: Step 4 durable analysis artifact.

Source idea: `ideas/open/192_residual_route_view_consumer_migration_map.md`

## Purpose

This map records the residual prepared consumers that remain after the
selected-consumer Route 1-7 migrations from ideas 182-188, plus the one x86
Route 6 interface reuse point from idea 189. It is intended to let future plan
owners create separable implementation ideas without re-deriving the residual
consumer inventory.

The selected readers are bounded prior art. They do not prove route-wide
prepared API retirement, `PreparedFunctionLookups` contraction,
`PreparedBirModule` deletion, or printer/debug/oracle replacement.

## Printer, Debug, And Oracle Consumers

Prepared printer, debug, route-debug, and oracle/test consumers remain
first-class residual consumers. They block prepared API contraction even when a
production reader has a route-first path, because they are the comparison
surfaces that prove route/prepared equivalence during migration.

| Surface | Current role | Contraction prerequisite |
| --- | --- | --- |
| Prepared-printer select-chain, producer/constant, publication, edge/source-producer, call, and memory displays | Printer/debug consumers for prepared facts and residual diagnostics. | Route-native diagnostics with equivalent content plus tests that preserve the prepared output contract until the replacement is proven. |
| Route-debug summaries and x86 debug/handoff output | Target-wrapper and debug validation for route/prepared agreement across wrapper boundaries. | A proven route-view interface for the matching route on AArch64, then target-wrapper proof that x86 or riscv consumes the same semantic interface without moving target-local policy into BIR. |
| `backend_prepared_lookup_helper`, block-entry publication, current-block join routing, fused-compare/materialized-condition, memory/frame/stack, scalar-record, call-boundary, and x86 handoff tests | Oracle/test consumers that compare prepared answers, fail-closed route behavior, and fallback behavior. | Replacement oracle coverage that checks the same semantic positives and negatives. These tests must not be weakened to make a route map look closer to retirement. |
| Prepared fallback branches inside production consumers | Temporary migration oracles for absent, mismatched, ambiguous, incomplete, or policy-sensitive route facts. | Per-consumer route/prepared equivalence proof over nearby same-feature cases, with the fallback still available until every residual consumer in that helper family is accounted for. |

## Target-Wrapper And Retained-Policy Consumers

The residual map separates target-neutral semantic route facts from target,
ABI, layout, address, move, and final-emission policy. These consumers should
not be converted by implication in a Route 1-7 semantic migration idea.

| Area | Retained consumers and policy | Migration boundary |
| --- | --- | --- |
| AArch64 traversal and aggregate prepared context | `PreparedBirModule`, `PreparedFunctionLookups`, call-plan, address-materialization, value-home, and move-bundle lookup threading. | Thread narrow route views beside prepared context only for selected semantic readers; do not create a BIR-owned clone of the prepared aggregate. |
| Calls and ABI | Call argument/result plans, lane bindings, variadic FPR counts, clobber/preserve sets, byval handling, outgoing stack sizing, helper/carrier protocols, late publication, and call-record spelling. | Route 6 may expose semantic call-use source facts one role class at a time; ABI placement and call record construction remain target/prepared policy. |
| Memory and addressing | Address bases, offsets, frame/stack objects, TLS/global relocation, pointer materialization, base-plus-offset legality, volatile/address-space handling, and final memory operands. | Route 3 may expose memory/source identity only. Prepared memory/addressing helpers remain policy owners and oracles. |
| Publication, homes, storage, and moves | Value homes, storage encodings, register choice, rematerialization spelling, move planning, move bundles, publication records, block order, and final machine records. | Routes 1, 4, and 5 may expose semantic producer/publication/edge facts; home, storage, scheduling, and emitted-record policy remain prepared/AArch64-owned. |
| Comparison, branch, and ALU policy | Fused legality, condition-code selection, branch spelling, hazards, emitted-register state, ALU result storage, final branch/compare record order, and materialized-compare render policy. | Route 7 may expose provenance only; branch and render policy remain target-owned. ## Future Idea Granularity

Future implementation ideas should be one route and one selected consumer
wide. Each idea should name:

- the exact production reader or wrapper boundary to migrate;
- the route record, typed index, validated facade, or compatibility adapter it
  will read;
- the prepared helper that remains public as fallback and oracle;
- the policy surfaces it refuses to migrate;
- route-present, route-absent, mismatched, ambiguous, and nearby same-feature
  proof expectations.

This artifact is complete for residual consumer classification and candidate
selection, but it is not a Phase E retirement plan. Draft 155 still needs a
field-by-field `PreparedBirModule` and `PreparedFunctionLookups` ownership map
before any prepared API contraction can be considered.
