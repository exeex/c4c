# AArch64 F128 Legacy Surface

This artifact preserves the useful production shape from the removed
`f128.cpp` translation surface. The old file was not active C++; it was a
line-commented mirror of the reference Rust implementation at
`ref/claudes-c-compiler/src/backend/arm/codegen/f128.rs`.

## Role

The surface implemented the AArch64 side of a shared binary128 soft-float
abstraction for `ArmCodegen`.

AArch64 `long double` was treated as IEEE 754 binary128, carried in 16-byte
Q registers for ABI-facing values. The backend expected no hardware quad-FP
arithmetic, so operations, comparisons, and conversions crossed through
compiler-rt or libgcc helper calls.

The shared orchestration lived outside this surface in
`backend/f128_softfloat.rs`; this file supplied architecture-specific storage,
register movement, address materialization, helper-call emission, and result
bridging.

## Entry Points

The old surface implemented `F128SoftFloat` for `ArmCodegen` and exposed three
small `ArmCodegen` helpers.

State and source lookup hooks:

- `state()`: returns the mutable codegen state.
- `f128_get_slot(val_id)`: looks up a stack slot for a value.
- `f128_get_source(val_id)`: retrieves tracked F128 source metadata.
- `f128_resolve_slot_addr(val_id)`: resolves a slot address for reloads.
- `f128_is_alloca(val_id)`: reports alloca-backed values.

Argument, stack, and address transport:

- `f128_load_const_to_arg1(lo, hi)`: loads the low and high 64-bit halves into
  `q0`.
- `f128_load_16b_from_addr_reg_to_arg1()`: loads `[x17]` into `q0`.
- `f128_load_from_frame_offset_to_arg1(offset)`: loads a 16-byte stack value
  into `q0`.
- `f128_load_ptr_to_addr_reg(slot, val_id)`: materializes an address in `x17`
  from an alloca or pointer slot.
- `f128_add_offset_to_addr_reg(offset)`: adds a small immediate or large
  scratch-loaded offset to `x17`.
- `f128_alloca_aligned_addr(slot, val_id)`: delegates aligned alloca address
  materialization.
- `f128_move_aligned_to_addr_reg()`: copies the alloca-aligned address register
  `x9` into the F128 address register `x17`.
- `f128_load_indirect_ptr_to_addr_reg(slot, val_id)`: loads an indirect pointer
  into `x17`.

Soft-float call and temporary-space hooks:

- `f128_load_operand_and_extend(op)`: materializes an operand through `x0`,
  moves it to `d0`, and calls `__extenddftf2`.
- `f128_move_arg1_to_arg2()`: copies `q0` to `q1`.
- `f128_save_arg1_to_sp()` and `f128_reload_arg1_from_sp()`: spill and reload
  `q0` at `[sp]`.
- `f128_alloc_temp_16()` and `f128_free_temp_16()`: reserve and release one
  16-byte temporary slot.
- `f128_call(name)`: emits a branch-and-link to the selected helper.
- `f128_truncate_result_to_acc()`: calls `__trunctfdf2` and moves `d0` raw bits
  into `x0`.

Store and copy hooks:

- `f128_store_const_halves_to_slot(lo, hi, slot)`: stores a constant low/high
  pair into a stack slot.
- `f128_store_arg1_to_slot(slot)`: stores `q0` to a stack slot.
- `f128_copy_slot_to_slot(src_offset, dest_slot)`: copies a 16-byte stack value
  through `q0`.
- `f128_copy_addr_reg_to_slot(dest_slot)`: copies `[x17]` to a stack slot.
- `f128_store_const_halves_to_addr(lo, hi)`: writes a constant low/high pair
  through the address in `x17`.
- `f128_save_addr_reg()`: saves `x17` into `x16`.
- `f128_copy_slot_to_saved_addr(src_offset)`: stores a stack F128 value through
  saved address `x16`.
- `f128_copy_addr_reg_to_saved_addr()`: copies `[x17]` to `[x16]`.
- `f128_store_arg1_to_saved_addr()`: stores `q0` through saved address `x16`.

Scalar result and cast hooks:

- `f128_flip_sign_bit()`: toggles the high-half sign bit of `q0`.
- `f128_cmp_result_to_bool(kind)`: maps an integer compare helper result in
  `w0` to a scalar boolean in `x0`.
- `f128_store_acc_to_dest(dest)`: stores scalar accumulator `x0`.
- `f128_track_self(dest_id)`: records that a value's full F128 representation
  lives in its own slot.
- `f128_set_acc_cache(dest_id)`: marks `x0` as the cached accumulator.
- `f128_set_dyn_alloca(val)`: temporarily changes dynamic alloca state.
- `f128_move_callee_reg_to_addr_reg(val_id)`: moves an assigned callee-saved
  register into `x17` when available.
- `f128_load_from_addr_reg_to_acc(dest)`: loads `q0` from `[x17]`, truncates to
  F64, moves raw `d0` bits to `x0`, and stores the scalar destination.
- `f128_load_from_direct_slot_to_acc(slot)`: loads a direct stack F128 value
  into `q0`.
- `f128_store_result_and_truncate(dest)`: stores the full `q0` result to the
  destination slot, tracks full-precision ownership, then leaves an F64
  approximation in `x0`.
- `f128_move_acc_to_arg0()` and `f128_move_arg0_to_acc()`: no-op on AArch64
  because the scalar accumulator and first argument are both `x0`.
- `f128_load_operand_to_acc(op)`: materializes an operand into `x0`.
- `f128_sign_extend_acc(from_size)` and `f128_zero_extend_acc(from_size)`:
  normalize narrow integer values in `x0`.
- `f128_narrow_acc(to_ty)`: delegates integer narrowing from `I64`.
- `f128_extend_float_to_f128(from_ty)`: bridges F32/F64 raw bits into `s0` or
  `d0` and calls `__extendsftf2` or `__extenddftf2`.
- `f128_truncate_to_float_acc(to_ty)`: calls `__trunctfsf2` or `__trunctfdf2`
  and moves raw FP result bits back to `w0` or `x0`.

Public helper shims:

- `emit_f128_operand_to_q0_full(op)`: delegates operand materialization to
  `f128_softfloat::f128_operand_to_arg1`.
- `emit_f128_store_q0_to_slot(slot)`: stores `q0` into a stack slot.
- `emit_f128_neg_full(dest, src)`: delegates full-precision negation to
  `f128_softfloat::f128_neg`.

## Register Conventions

- `q0` / `v0`: primary F128 argument and result carrier.
- `q1` / `v1`: secondary F128 argument carrier.
- `d0` / `s0`: F64/F32 helper-call argument or return register.
- `x0` / `w0`: scalar accumulator, integer argument, helper result, and raw
  FP-bit bridge.
- `x1`: high half of an immediate F128 constant before insertion into `q0`.
- `x16`: scratch register and saved-address carrier.
- `x17`: active F128 address register.
- `x9`: alloca-aligned address register copied into `x17` when needed.
- `sp`: temporary spill base for one 16-byte F128 value.

## Binary128 Representation

The old route treated every F128 value as 16 bytes. Stack slots, temporary
space, and all memory copies used 16-byte transfers. Constants were expressed
as two 64-bit halves, with the low half stored at the base address and the high
half stored at base plus eight bytes.

Q-register construction from halves used:

```asm
fmov d0, x0
mov v0.d[1], x1
```

Full F128 memory transport used `ldr q0, [...]` and `str q0, [...]` whenever
the address was known. The architecture-specific address register for shared
F128 orchestration was `x17`.

## Helper Calls And FP Bridges

The surface expected arithmetic and conversions to be implemented by external
soft-float helpers. Visible helper names included:

- `__extenddftf2`
- `__extendsftf2`
- `__trunctfdf2`
- `__trunctfsf2`

Generic calls were emitted by name through `f128_call(name)`, letting the
shared orchestration choose arithmetic, comparison, or conversion helper
symbols.

F64 and F32 values were carried in integer registers as raw bits until crossing
into helper-call FP registers:

- F64 input bridge: `fmov d0, x0`
- F32 input bridge: `fmov s0, w0`
- F64 output bridge: `fmov x0, d0`
- F32 output bridge: `fmov w0, s0`

Helper calls that could clobber caller-saved registers invalidated the register
cache before later operand materialization reused cached values.

## Source Tracking And Full-Precision Reloads

The old comments called out `f128_load_sources` metadata as a key design point.
The implementation exposed `get_f128_source`, `resolve_slot_addr`, and
`track_f128_self` hooks so shared orchestration could reload full 16-byte
values for comparisons, casts, loads, stores, and negation instead of relying
only on a truncated scalar approximation.

`f128_store_result_and_truncate` was the clearest example of the dual contract:
it stored the full `q0` result to the destination slot and tracked that full
source first, then produced an F64 approximation in `x0` for scalar accumulator
compatibility without overwriting the 16-byte slot.

## Dependencies

The removed surface depended on these surrounding concepts:

- IR model: `Operand`, `Value`, and `IrType`.
- Backend state: `CodegenState`, `StackSlot`, `SlotAddr`, stack-slot lookup,
  alloca detection, F128 source tracking, dynamic alloca state, and register
  cache state.
- AArch64 codegen helpers: immediate loading, stack load/store emission, stack
  adjustment, alloca address emission, aligned alloca address emission, scalar
  operand materialization, scalar destination storage, cast instruction
  emission, and callee-saved register naming.
- Shared binary128 orchestration in `backend/f128_softfloat`.
- Cast comparison kind mapping through `backend::cast::F128CmpKind`.
- Runtime/compiler-rt helper availability for binary128 conversion,
  arithmetic, comparison, and negation paths.

## Hidden Assumptions

- AArch64 binary128 values are ABI-carried in Q registers and are exactly
  16 bytes in stack slots.
- The shared soft-float layer owns operation sequencing; this surface only
  supplies AArch64-specific primitives.
- `x17` is available as the active F128 address register across shared helper
  sequences.
- `x16` is available as scratch and as a saved-address register when a copy
  needs both source and destination addresses.
- All `q0` stack spills reserve exactly 16 bytes and preserve alignment
  expectations required by `ldr q` and `str q`.
- Integer and F32/F64 values travel through `x0`/`w0` as raw bits until an
  explicit `fmov` bridge.
- Calls to soft-float helpers may clobber caller-saved registers; cached
  operands must be invalidated after those calls.
- `f128_store_result_and_truncate` must not store the truncated F64
  approximation back into the F128 destination slot.
- `f128_cmp_result_to_bool` assumes helper comparison results are signed
  integers in `w0` where zero means equality and signed ordering maps to
  AArch64 `lt`, `le`, `gt`, and `ge` conditions.

## Rebuild Risks

- Losing full-source tracking would make later comparisons and casts reload an
  F64 approximation instead of the full binary128 payload.
- Reusing `x16` or `x17` for unrelated scratch work inside shared sequences
  would corrupt indirect F128 loads and stores.
- Treating binary128 negation or casts as scalar F64 operations would silently
  discard precision.
- Forgetting register-cache invalidation after helper calls could skip
  necessary reloads after caller-saved registers are clobbered.
- Storing the truncated scalar accumulator into an F128 destination slot would
  overwrite the full 16-byte result.
- Assuming hardware quad-FP arithmetic exists on AArch64 would bypass the
  required soft-float helper boundary.
