### Selected First Extraction Target

Select a shared target-neutral prepared value-home query used by `dispatch_publication.cpp::prepared_value_home_for_value` and `dispatch_lookup.cpp::make_named_prepared_result_register`.

Rationale:

- The shared data owner already exists in `src/backend/prealloc/value_locations.hpp`: `PreparedValueHomeLookups`, `find_indexed_prepared_value_id`, and `find_indexed_prepared_value_home` express the fact relationship without target policy.
- The AArch64 wrappers duplicate the same local rediscovery pattern: resolve `bir::Value` name through prepared names, then query value ID/home using `value_home_lookups`, `regalloc`, and `value_locations`.
- The extraction can be narrow: add a shared query that returns `const PreparedValueHome*` for a prepared-name table plus BIR value and the existing lookup sources. Keep AArch64-specific register-bank parsing, scalar register view selection, stack/frame-pointer materialization, scratch behavior, and `RegisterOperand` construction in AArch64 code.
- Nearby same-feature cases are abundant enough to avoid a one-case wrapper: `prepared_value_home_for_value` call sites in `dispatch.cpp`, `dispatch_value_materialization.cpp`, `dispatch_edge_copies.cpp`, `comparison.cpp`, `globals.cpp`, `memory.cpp`, `calls.cpp`, `cast_ops.cpp`, `variadic.cpp`, and `fp_value_materialization.cpp`; `make_named_prepared_result_register` call sites in `globals.cpp`, `memory.cpp`, `calls.cpp`, `alu.cpp`, `comparison.cpp`, and `memory_store_retargeting.cpp`.

## Watchouts

- Do not move `abi::parse_aarch64_register_name`, `scalar_register_view`, `scalar_view_for_type`, `gp_register_name`, `RegisterOperand` construction, stack/frame-pointer decisions, or scratch/hazard policy into shared code.
- The shared query should not perform string matching beyond resolving a BIR named value through `PreparedNameTables`; it should reuse interned `ValueNameId` and existing value-home lookup helpers.
- `current_block_entry_publication_*` and same-block producer/select-chain helpers are valid follow-ups, but they are larger and should not be mixed into the first value-home extraction unless a compile boundary requires a tiny helper signature adjustment.
- Keep shared code target-neutral; AArch64 register choices, scratch handling, hazard policy, relocation operands, and final instruction spelling must remain local.
- Do not replace prepared facts with predecessor rescans, BIR-name matching, or local reconstruction behind a generic API name.
- Do not claim progress from helper renames, expectation rewrites, or a one-case wrapper.
- Treat ideas `134` and `135` as separate open follow-ups unless the selected 133 packet directly needs the same fact relationship.
