## Just Finished

Completed `plan.md` Step 2 trace for the `src/20080506-2.c` frame-slot
address call argument.

Implementation sites:

- Semantic source selection starts in
  `src/backend/bir/lir_to_bir/calling.cpp`: local pointer call arguments are
  annotated as `CallArgumentSourceSelectionKind::FrameSlotAddress` when the
  named argument is a local slot.
- Prepared selection is populated in
  `src/backend/prealloc/call_plans.cpp` by
  `select_prepared_call_argument_source`; for `source_encoding=FrameSlot`, a
  latest frame-slot address materialization makes the argument
  `PreparedCallArgumentSourceSelectionKind::FrameSlotAddress`.
- The missing marker is generated by
  `find_prepared_missing_frame_slot_call_argument_publication_need` in
  `src/backend/prealloc/calls.hpp`. This is a need classifier for frame-slot
  arguments destined to one GPR, not a proof that no payload fact exists.
- The payload fact is generated by
  `populate_call_argument_value_publication_plans` in
  `src/backend/prealloc/publication_plans.cpp`, which selects a unique earlier
  store-local publication into the argument object slot.
- Store-local publication source facts are built by
  `plan_prepared_store_source_publication` and
  `populate_store_source_publication_plans` in
  `src/backend/prealloc/publication_plans.cpp`. For the representative, the
  source is recorded as `payload=%lv.a` but with
  `source_producer=unknown`, so the fact carries the payload value, not a
  frame-slot-address payload route.
- RV64 object call lowering consumes the fact in
  `src/backend/mir/riscv/codegen/object_emission.cpp` through
  `prepared_frame_slot_address_call_argument_publication` and the
  `FrameSlotAddress` argument branch.
- RV64 text call lowering in
  `src/backend/mir/riscv/codegen/prepared_call_emit.cpp` only tries
  `emit_riscv_frame_slot_address_argument` under the register-source branch;
  a `source_encoding=FrameSlot` / `FrameSlotAddress` argument falls through to
  generic argument materialization instead of the publication fact route.

Current wrong rule:

- Prepared BIR has the needed arg1 object-publication fact, but that fact does
  not publish that the payload `%lv.a` is itself a frame-slot address. RV64
  object lowering therefore materializes `payload=%lv.a` with generic value-home
  code, producing the stale edge `mv t1,s1`; `sd t1,16(sp)` before
  `addi a1,sp,16`.
- The prepared move reason `call_arg_stack_to_register` describes the regalloc
  spill slot for `%lv.p2`; it is not sufficient authority for the desired
  value stored into the `%lv.p2` object slot. The desired value is the published
  payload address of `%lv.a`.

## Suggested Next

Execute Step 3 by repairing both sides of the route:

- Extend prepared call-argument value publication so the selected store-local
  payload can carry frame-slot-address authority for payload values like
  `%lv.a` when the store source is a local object address.
- Make RV64 frame-slot call-argument consumption use that published payload
  authority directly to write the pointee object slot, then materialize the
  argument object address into the ABI destination register.
- Add the same direct consumption to the RV64 text route or fail closed there;
  do not leave `source_encoding=FrameSlot` / `FrameSlotAddress` to generic
  `emit_move_to_register`.

  and report the producer gap instead of reconstructing it in RV64.
- Step 1 found payload authority for arg1, so Step 2 should distinguish the
  missing publication marker from the existing payload record instead of
  treating the source fact as absent.
- Keep `missing_frame_slot_arg_publication=yes` as a need signal until the
  producer publishes a coherent payload route; do not suppress the marker as a
  cosmetic fix.
- Fail closed if the selected publication is ambiguous, lacks a unique
  store-local source, has a non-default/TLS/volatile destination, has a payload
  value equal to the argument object value, or cannot prove the payload is a
  materializable local frame-slot address.
- Do not reconstruct `%lv.a` from the testcase source, variable names, the
  destination slot number, or qemu signal status in RV64. RV64 should consume a
  prepared payload route.
