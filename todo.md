## Watchouts

- `traversal.cpp` intentionally builds one `PreparedFunctionLookups` aggregate
  for AArch64 lowering. Splitting owner files should not remove the ability to
  build coherent per-function lookup state cheaply.
- AArch64 `calls.cpp` is a large consumer for many domains, but most uses still
  read semantic prepared facts before target-local register/stack emission.
  Treat size and call count as routing evidence, not proof of AArch64-shaped
  shared API.
- The strongest concrete Step 3 smell is duplicate select-chain public
  exposure across `prepared_lookups.hpp` and `publication_plans.hpp`.
- No API was unmapped. The main ambiguity is ownership, not consumer discovery,
  for stack/frame id helpers, call-plan indexes, memory/address indexes, and
  edge-publication fact planners.
